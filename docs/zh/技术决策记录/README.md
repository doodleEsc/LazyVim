# 技术决策记录 (Architecture Decision Records)

*LazyVim 项目重要技术决策的记录和追踪*

---

## 什么是ADR

Architecture Decision Records (ADR) 是记录重要技术决策的轻量级文档格式。它们帮助团队：

- 📝 **记录决策背景** - 为什么需要做出这个决策
- ⚖️ **对比方案选择** - 考虑了哪些替代方案
- 🎯 **明确最终决策** - 选择了什么方案及其理由
- 🔄 **追踪决策演进** - 决策如何随时间变化

---

## ADR 编号系统

LazyVim 使用连续编号系统：

```
001-选择lazy.nvim作为插件管理器.md
002-采用分层配置覆盖机制.md
003-引入Extra插件系统.md
...
```

**编号规则**：
- 使用3位数字前缀（001, 002, 003...）
- 中文标题，简洁描述决策内容
- 文件名使用kebab-case（用连字符分隔）
- 按时间顺序递增编号

---

## 创建新ADR

### 1. 选择编号
查看现有ADR文件，选择下一个可用编号。

### 2. 使用模板
复制 `ADR-模板.md` 文件，重命名为新的ADR编号。

### 3. 填写内容
根据模板填写各个部分，确保信息完整。

### 4. 提交审查
创建Pull Request，邀请团队成员审查决策记录。

## ADR 状态

每个ADR都有明确的状态标识：

- **🎯 提议 (Proposed)** - 决策正在讨论中
- **✅ 已接受 (Accepted)** - 决策已被采纳并实施
- **❌ 已拒绝 (Rejected)** - 决策被拒绝，不会实施
- **⚠️ 已废弃 (Deprecated)** - 决策已过时，被新决策替代
- **🔄 已替代 (Superseded)** - 被更新的ADR替代

## 最佳实践

### 何时创建ADR

**应该创建ADR的情况**：
- 🏗️ 重要架构决策（如插件管理器选择）
- 📊 性能优化策略（如懒加载机制设计）
- 🔧 工具链选择（如代码格式化工具）
- 🎯 用户体验决策（如配置覆盖机制）
- 🔒 安全相关决策（如敏感信息处理）

**不需要创建ADR的情况**：
- 🐛 日常bug修复
- 📝 文档更新
- 🎨 代码重构（不改变行为）
- 🔧 小的配置调整

### 编写指南

1. **简洁明了** - 控制在2-4页内，重点突出
2. **客观描述** - 避免主观判断，基于事实和数据
3. **考虑全面** - 列出主要替代方案及其权衡
4. **可操作性** - 明确决策的具体实施步骤
5. **可追溯** - 包含相关讨论和参考资料链接

### 维护更新

- **定期审查** - 每季度检查ADR的有效性
- **状态更新** - 及时更新过时或已替代的ADR状态
- **交叉引用** - 新ADR应引用相关的旧ADR
- **影响评估** - 重大变更时评估对现有ADR的影响

## 贡献指南

### 创建ADR

1. **识别需求** - 确认决策的重要性和影响范围
2. **调研背景** - 收集相关信息和约束条件
3. **分析方案** - 列出可行的替代方案
4. **评估权衡** - 分析各方案的优缺点
5. **形成决策** - 选择最佳方案并说明理由
6. **编写记录** - 使用模板创建ADR文档
7. **团队审查** - 通过PR进行同行审查
8. **确认接受** - 团队达成共识后标记为"已接受"

### 审查要点

**内容审查**：
- [ ] 问题描述清晰准确
- [ ] 替代方案考虑充分
- [ ] 决策理由充分合理
- [ ] 实施后果分析到位

**格式审查**：
- [ ] 使用标准模板格式
- [ ] 编号和标题规范
- [ ] 状态标识正确
- [ ] 相关链接有效
